JAVA编程思想：二十-注解

Posted on 2019-02-24 Edited on 2023-03-28

注解(也被称为元数据)，为我们在代码中添加信息提供了一种形式化的方法，使我们可以在稍后某个时刻非常方便的使用这些数据

注解使得我们能够以将由编译器来测试和验证的格式，存储有关程序的额外信息，注解可以用来生成描述符文件，甚至或是新的类定义，并且有助于减轻编写“样板”代码的负担

java SE5内置了三种，定义在java.lang中的注解（标准注解）

  • @Override
  • @Deprecated
  • @SuppressWarnings

1.基本语法

注解的定义很像是接口的定义，事实上与其他任何java接口一样，注解也会编译成class文件

在注解中一般会包含某些元素以表示某些值，当分析注解时，程序或工具可以利用这些值，没有元素的注解称为标记注解

有元素注解示例：

在定义注解时，会需要一些元注解(meta-annotation),如@Target和@Retention

  • @Target用来定义你的注解将应用于什么地方（例如一个方法或是一个域）,ElementType可选值包括：

    • CONSTRUCTOR：构造器的声明
    • FIElD：域声明（包括enum元素）
    • LOCAL_VARIABLE：局部变量声明
    • METHOD：方法声明
    • PACKAGE：包声明
    • PARAMETER：参数声明
    • TYPE：类、接口（包含注解类型）或enum声明

    如果想要将该注解应用于所有的ElementType，那么可以省去@Target元注解，不过这不常见

  • @Retention用来定义该注解在哪一个级别可用，

    • SOURCE：在源代码中，注解被编译器丢弃
    • CLASS：类文件中，注解在class文件中可用，但会被jvm丢弃
    • RUNTIME：运行时，JVM在运行期也保留注解，因此可以通过反射机制读取注解信息

  • @Decumented：将次注解包含在javadoc中

  • @Inherited：允许子类继承父类中的注解

大多数情况，程序员自己定义注解，并编写自己的处理器来处理他们

2.编写注解处理器

在使用注解的过程中，很重要的一个部分就是创建与使用注解处理器

一般可使用反射获取注解并对注解进行处理

注解元素的可用类型如下：

  • 所有基本类型（int、float、boolen等）
  • String
  • Class
  • enum
  • Annotation
  • 以上类型的数组

使用其他类型，编译器即将报错（特殊注意的是：也不允许使用任何包装类型）

对于注解的元素：不能有不确定的值，也就是说，元素要么具有默认值，要么在使用注解时提供元素的值

其次对于非基本类型的元素，无论是在源代码中声明时，或是在注解接口中定义默认值时，都不能以null作为其值，这个约束使得处理器很难表现一个元素的存在或是缺失的状态

  • 为类绕开这个约束，我们只能自己定义一些特殊的值，例如空字符串或负数，以表示某个元素不存在。在定义注解的时候，这算是一个习惯用法

如果程序员的注解中定义了名为value的元素，并且在应用该注解的时候，该元素是唯一需要赋值的元素，那么此时无需使用名-值对的这种写法，而只需在括号内给出value元素所需的值即可，这可以应用于任何合法类型的元素，当然这也限制了程序员必须将此元素命名为value,不过这样可能更加简洁

使用对个注解的时候，同一个注解不能重复使用

注解不支持继承

  • 不能使用关键字extends来继承某个@interface

3.使用apt处理注解

apt是注解处理工具

apt与javac一样被设计为操作java源文件

使用apt生成注解处理器时，我们无法利用java的反射机制，因为我们操作的是源代码，而不是编译后的类

4.基于注解的单元测试

在java中，最著名的单元测试就是JUnit,JUnit中就使用了注解

5.总结

注解是java引入的一项非常受欢迎的补充

他提供了一种结构化的，并且具有类型检查能力的新途径，从而使得程序员能够为代码加入元数据，而不会导致代码杂乱且难以阅读，使用注解能够帮助我们避免编写累赘的部署描述文件，以及其他生成的文件，而javadoc中的@deprecated被@Deprecated注解取代的事实也说明，与注释性文字相比，注解绝对更适合用于描述类相关的信息
